# GenbaHub 再構築設計書 — 「使い始めたら離せない」への道

作成: 2026-07-20 Fable設計者 / 発注: 光輝さん「UI,UX使い心地・軽さ・AIフレンドリーに全パーツが繋がる設計で再構築、公開できるレベルに」

## 結論（30秒版）

1. **作り直しは不要。「配線のやり直し」が必要。** 機能量は既に大手級（47ページ+20機能画面、141K行）。問題は (a) データ経路が3系統に分裂しUIとAIが別世界を見ている、(b) 機能過多でナビと初回体験が渋滞、(c) 品質の信頼を壊す「画面間で数字が食い違う」既知バグ群。
2. **最重要の一手 = 統一コマンド層。** UI・MCP・REST・AI秘書の全てを同じコマンド関数群（create_project / add_task / attach_photo / issue_invoice...）に通す。これが「AIフレンドリーに全てのパーツが繋がる」の実体であり、以後の全機能が自動的にAI操作可能になる。
3. **北極星指標: 初回セッション3分以内に「案件1件+工程1本+写真1枚」が完了し、翌日も開く。** 導線・軽さ・オンボーディングの全判断をこの指標で裁く。

## 0. 北極星と代理指標

「使い始めたら離せない」は直接測れないので、以下の代理指標で運用する:

| 指標 | 目標 | 現状実測 |
|---|---|---|
| TTV (Time to Value): 初回起動→案件1+工程1+写真1完了 | **3分以内** | デスクトップは到達可能圏（案件2タップ/工程3タップ）だが、**モバイルは工程登録がバグで完了不能・写真は導線バグ**（§2.2） |
| 主要5タスクのタップ数（案件/工程/写真/見積/請求） | 全フロー完走可能+各≤4タップ | 2 / 3(モバイル不能) / 未実測 / 4 / **導線なし(AR請求が機能欠落)** |
| 初期表示 (cold): 初期ロードJS+CSS | **< 200KB gzip** | **390KB gzip**（うちjspdf/canvg漏れ分が主犯、§2.3） |
| AI操作カバレッジ: UI機能のうちAIから実行可能な割合 | **100%（コマンド層経由）** | **約27%**（15ドメイン中4、金流は全滅、§2.4） |
| 「数字の矛盾」既知バグ | **0件**（bnu/qgm/e0q） | 3件オープン |

「離せない」の実体は毎朝開く理由 = **「今日」画面が現場の1日を数字で正しく言い当てること**。数字が画面間で矛盾している限りリテンションは立たない（scorecard-20260707 #10/#11参照）。機能追加より矛盾ゼロが先。

## 1. 現状監査（実物ベース）

### 1.1 構造

- Vite + React 19 SPA、hashルーティング自作（`src/hooks/useHashRouter.ts`）、`src/App.tsx`(1,256行) に全ルート定義75分岐+lazy 64本。react-router不使用。
- ページ: `src/pages/` 47 + `src/components/` 内ページ級 ~20（MarginWatch/CrewOptimizer/LocalSeo等、ナビ未露出のseedデモ含む=既知票 k58）。
- 一次ナビは直近の bc4983a で 今日/案件/工程/写真 の4タブに整理済み（正しい方向）。ただし field/money グループに18項目が残existり、二次ナビはまだ「全部盛り」。
- 依存は健全（chart库なし、重いのは pdfjs-dist / framer-motion / jspdf / supabase-js）。

### 1.2 データ経路が5系統に分裂している（最大の構造問題・実配線監査で確定）

| 経路 | 実体 | 稼働状態 | 使用者 |
|---|---|---|---|
| A. UIフロント直結 | `src/infra/create-app-repository.ts` → Supabase(anon+RLS) or localStorage | **本番稼働** | 全ページ（`src/stores/*` / `lib/supabase-adapter` 30リポジトリ経由。fetchで/apiを叩くページ**ゼロ**を3ページ実読で確認） |
| B. REST API | `src/api/server.ts` + `routes/`（~30エンドポイント） | **非デプロイ**（Vercelに配線なし、`pnpm api:server`手動起動のみ） | CLI (`genbahub-cli`) |
| C. リポジトリ内MCP | `src/mcp/genbahub-mcp-server.ts`（27ツール、service_role直結） | **死に配線**（`~/.claude.json`から参照されていない。MCP_SETUP.mdはこちらを案内=doc乖離） | 誰も使っていない |
| D. 実稼働MCP | **別リポジトリ `~/genbahub-mcp/`**（14ツール+nl_dispatch、Supabase直叩き） | **稼働中** | AI（社内bot）。estimates/reports/schedule系ツールは無い |
| E. Vercel functions | `./api/*.ts`（freee/OCR/Stripe/chat/push/scan-import） | 本番稼働 | 決済・会計・中継専用。BはおろかA とも独立実装 |

帰結:
- **同じ「案件作成」がA/B/C/Dの4実装**あり、snake/camel変換・バリデーションが各々別コード。画面間の数字矛盾（bnu/qgm/e0q）と同根の構造。
- C の `report-tools.ts` は **`new Map()` のプロセス内メモリ保存**（Supabaseに書かれず再起動で消える偽データ）。「動くように見えてどこにも繋がっていない」の典型。
- C/D とも service_role 級の直叩きでマルチテナント化と両立しない（RLS票 tku/791 と衝突）。
- 実装済みなのにAIから届かない資産が大量にある: B の27 RESTルート、C の見積NLパース/日報/MEO系13ツール。

さらに **in-app AI秘書は Discord中継**（`api/chat/[action].ts` が bot ID 1488015940565209148 のDiscordチャンネルをポーリング）。社内デモとしては動くが、公開SaaSの機能としては成立しない（他社テナントが光輝さんの個人botに相談する構造）。ここも統一コマンド層+サーバ側エージェントに置換が必要。

### 1.3 導線（初回体験）

- 設計上は AuthGuard → OnboardingWizard 4ステップ（1画面3ボタン原則、docs/onboarding-flow.md）→ サンプル案件生成、の流れ。**しかし実挙動は `ensureFirstRunProject` が確認なしにサンプル案件を生成して /gantt へ強制リダイレクトし、Wizardを見る余地がない**（§2.2実測）。設計docと実装が乖離。
- 実測タップ数: §2.2

### 1.4 軽さ

- lazy 64本でページ分割自体は済んでいる。問題はエントリ側（App.tsx 1,256行+i18n+framer-motion+全nav定義が初期同梱）と、PWA SWが旧バンドルを配り続ける既知票 88n（体感の遅さ・古さの正体になり得る）。

### 2.3 軽さ実測

| 項目 | 実測値 |
|---|---|
| 総JS（123チャンク+pdf.worker） | raw 6,166 kB / gzip 2,134 kB（うちpdf.worker単体 raw 1,418 / gzip 416） |
| エントリ `<script>` 単体 (index-*.js) | raw 347.37 kB / **gzip 110.04 kB** |
| 初期ロード合計（entry+modulepreload 9本+CSS、`dist/index.html`実測） | raw 1,267.30 kB / **gzip 376.0 kB** ← 目標200KBの約1.9倍 |
| ビルド時間 | wall 2.4s（Vite 8/Rolldown、PWA precache 130 entries 4.74MB込み） |
| lazy分割 | 64本有効。**モノリスではない**（100+チャンクに分割済み） |

**確定した犯人（dist/index.html の modulepreload 実測で確認）**:
- 初回ロードに同乗している9チャンクのうち **jspdf 400.76 kB + canvg 151.76 kB ≒ 552 kB raw（gzip ~179 kB）**。原因は `src/estimate/pdf-estimate.ts:1` の `import { jsPDF } from "jspdf"`（静的import）。同リポの他呼び出し（report-generator.ts:97 / site-entry-qr.ts:195）は正しく `await import("jspdf")` にしており、**この1行だけの問題**。これを動的化すると初期gzip ≈ 197 kB となり目標200KBを一発達成。
- framer-motion がエントリ同梱（AssistantChatPanel が App.tsx で常時レンダーされるため）。
- 正しく遅延できているもの: pdf-font 1.27MB / pdfjs 374KB / supabase本体 184KB / html2canvas 200KB / GanttPage 157KB / ProjectDetailPage 156KB。
- PWA SW が 130エントリ 4.74MB を precache しつつ旧バンドルを配り続ける既知票 88n が「更新したのに遅い/古い」の隠れ犯人。

結論: 「軽さ」はアーキテクチャ問題ではなく **(1) jspdf静的import 1行、(2) framer-motion常駐、(3) SW更新戦略(88n)** の3点修正で北極星目標(<200KB gzip)に到達できる。

## 3. 再構築設計

### 3.1 統一コマンド層（最優先・「全てのパーツが繋がる」の実体）

```
UI(ページ/フォーム)   MCPサーバ   REST API   AI秘書(サーバ側)
        \            |           |           /
         └────→  src/commands/*  ←──────────┘
                  （唯一の書き込み経路・zodスキーマ・権限チェック・監査ログ）
                          |
                  Repository層（Supabase RLS / localStorage）
```

原則:
1. **書き込みは全てコマンド経由**。コマンドは `{ name, input(zod), execute(ctx), permission }` の純関数。UIのフォームsubmitもMCPツールも同じ関数を呼ぶ。
2. **MCPツールはコマンドの自動生成**にする（zodスキーマ→MCP tool定義は機械変換）。新機能=コマンド1個追加でUI/AI両対応が同時に手に入る。以後カバレッジ100%が構造的に保たれる。
3. **service_role廃止**: MCP/APIはユーザー(またはorg)スコープのトークンで RLS を通す。票 tku/791 と同一線上で解決。
4. 読み取りは既存Repository直読みを当面許容（段階移行）。書き込み統一が先。
5. 監査ログ（誰が/どの経路で/何を変えたか）をコマンド層で一括取得 → 「AIがやった変更」の可視化はSaaSの売りになる。

段階移行（ビッグバン禁止）:
- Phase 1: `src/commands/` 新設。案件/工程/写真/見積/請求の書き込み系コマンド10本を既存ロジックから抽出。UIは既存storeを残したままコマンドを内側で呼ぶ形に差し替え。
- Phase 2: MCPサーバをコマンド層の自動公開に置換。このとき**3つの死に配線を整理統合**する: リポジトリ内MCP(C)退役、別リポ `~/genbahub-mcp`(D) を本リポのコマンド層ベースに建て替え、非デプロイREST(B)は v1読み取りゲートウェイのみ残して退役。MCP_SETUP.md を実配線に一致させる。
- Phase 3: AI秘書をDiscord中継からサーバ側エージェント（コマンド層をtool群として持つ）へ置換。

### 3.2 導線の再設計

- **一次ナビ4タブは維持**（今日/案件/工程/写真）。二次ナビ18項目は「案件の中」へ畳む: 見積/請求/原価/発注/報告は案件詳細のタブにし、横断ビューは「今日」からのリンクに降格。原則: **現場の1日は案件単位で動く。横断機能は経営者の週次**。
- **グローバル「+」1個**: どの画面でも同じ位置の「+」→ 案件/工程/写真/見積 の4択。作成系導線をここに一本化（迷いの根絶）。
- **空状態=次アクション**: 全一覧の空状態に主ボタン1つ（onboarding-flow.mdの1画面3ボタン原則を全画面に拡張）。
- ナビ未露出のseedデモページ群（k58）は公開ビルドから除外（ルート自体を落とす）。

### 3.3 軽さ

目標: エントリJS < 200KB gzip / 初回表示(ローカル) < 1.5s / ルート遷移 < 300ms。
手段（実測を見て §2.3 の犯人から順に）:
1. App.tsx からルートテーブル・nav定義を分離し、i18n初期バンドルを日本語のみに
2. framer-motion / pdfjs / jspdf が エントリに漏れていないか実測で確認し、漏れていれば動的import境界を修正
3. SW更新戦略（票 88n）: skipWaiting+更新トースト。**旧バンドル配信は「重い」体感の隠れ犯人**
4. Supabase初期クエリの直列待ちを「今日」画面で並列化+スケルトン表示
