# ドキュメントタブ インポート/エクスポート GoodNotes級ギャップ表 (2026-07-07)

光輝さん指示（GenbaHub ch, 2026-07-07）「goodnotes並みのインポート、エクスポートとかになってるか？そこまでループで仕上げてデザインも拘って」への対応。ループ1周目。

検証環境: ローカルdev (`vite --port 5183`) + `window.__E2E_BYPASS_AUTH__`(LocalStorageRepository経路、本番Supabase非接触) + Playwright headless Chromium。デスクトップ(1440px)・モバイル(390px)を実機相当で確認。
検証コミット起点: `60ba517`（本ループの変更前）

## 現状監査（変更前）

- インポート経路: 「アップロード登録」フォームがファイル名/種別/**ファイルURL**の3項目を要求する **URL参照方式のみ**。ローカルファイルを選ぶ導線が存在せず、事前にGoogle Drive等へアップロードしてURLを貼る前提（`DocumentsPageImpl.tsx`のUI文言「ファイル本体は Drive 等に置き、ここでは参照URLを管理します」より意図的な設計と判明）。
- 一括登録: 不可。1件ずつフォーム送信。
- 即プレビュー: 送信直後に選択状態にはなるが、そもそもファイルをその場で選べないため「ドラッグ→即確認」体験がない。
- エクスポート経路: プレビュー右上の「元ファイルを開く」リンク（新規タブ）のみ。ダウンロードやOS共有シートへの導線は無し。
- ページ管理・注釈: 無し（想定通り、今回スコープ外の大物）。

## 実装した2件

### 1. ドラッグ&ドロップ一括インポート
- `src/components/DocumentsPageImpl.tsx`: ページ全体を drop target化。`dragenter`/`dragover`/`dragleave`/`drop` を実装し、複数ファイルを検証(`isSupportedDocumentFile`)→アップロード(`uploadProjectDocumentFile`)→`documentRepository.create()`まで一括処理。
- 種別は画像→「写真」、PDF等→「図面」に自動推定(`inferDocumentTypeFromFile`)。
- 新規: `src/infra/supabase-adapter/document-file-storage.ts`。写真アップロード(`photo-repository.ts`)と同系統のSupabase Storage実装。本番接続時は署名付きURLを発行、ローカル/E2Eバイパス時は`URL.createObjectURL`にフォールバック。
- 新規(draft): `supabase/migrations/draft_2026_07_07_project_documents_storage.sql`。`project-documents`バケット+RLSポリシー。**本番未適用**（後述）。
- 実装中に発見・修正した実バグ: `getPreviewConfig()`がURLの拡張子だけで形式判定していたため、拡張子を持たないblob:オブジェクトURL（D&D経由のファイル）はプレビュー対象と認識されず`window.open`フォールバックに落ちて「即プレビュー」が成立していなかった。元ファイル名を判定のフォールバックに追加して解消（回帰テスト付き）。

### 2. ワンタップ共有/エクスポート
- プレビュー右上に「共有 / ダウンロード」ボタンを追加。`fetch→blob→File`化し、`navigator.canShare`/`navigator.share`があればOS共有シートを1タップで起動。非対応時・fetch失敗時（CORS等）はダウンロードまたは新規タブ表示にフォールバック。

## 重要な限定事項（正直に書く）

1. **本番Supabaseへの実ファイル永続化は未適用**。今回追加した`project-documents`バケット定義はdraft migrationのみで、本番へは未適用（「本番DB直接適用禁止」の境界を厳守）。**現状のまま本番にデプロイした場合、Supabase接続あり・E2Eバイパス無しの本番環境ではD&Dアップロードがバケット不在エラーで失敗する。** 光輝さんの go-ahead を得て `draft_2026_07_07_project_documents_storage.sql` を適用するまでは、本番でのD&D実ファイル取り込みは機能しない。ローカル/E2E検証はこの限りではなく正常動作（オブジェクトURLへの意図的フォールバック）。
2. ローカル/E2Eフォールバックの`blob:`URLはブラウザタブ内でのみ有効（リロード・他端末では参照不可）。既存のLocalStorageRepository全体がそもそも本番相当ではない検証専用モードのため、この制約は新規リスクではなく既存の枠組みと同じ性質。
3. 大容量PDFのテキストはjsPDFで生成した検証用ダミーPDFで、日本語テキストが文字化けして描画されている（`07_large_pdf_page2_zoomed_desktop_1440.png`参照）。これはjsPDFの既定フォントがCJKグリフを含まないテスト用フィクスチャの制約であり、アプリ側(pdf.js描画)の不具合ではない。ページ送り・拡大・レンダリング完了自体は正しく動作している。

## 体感で一番ストレスだった点 / あれば作業が半分になった機能

1. **一番ストレスだった点**: Playwrightでファイルドロップを模擬する際、ブラウザの`DataTransfer`/`File`をNode側からそのまま渡せず、base64往復でbrowser context内に再構成する必要があった（`page.evaluateHandle`経由）。実機のD&D動作そのものより、**D&Dを自動テストで再現する手順の方が手間がかかった**。
2. **あれば作業が半分になった機能**:
   - Playwright/chrome-devtools系ツールに「ローカルファイルパスを渡すだけでdrop eventを発火してくれるヘルパー」があれば、上記のbase64往復が不要になった。
   - 既存コードに`isSupportedDocumentFile`相当の「対応ファイル形式ホワイトリスト」が既に散在(`InvoicePage.tsx`は`image/*,.pdf`、写真は`image/jpeg,png,heic,heif`)しており、今回また独自定義した。ドキュメント種別ごとの許容MIME/拡張子を1箇所にまとめた共通定数があれば重複実装を避けられた。
   - Supabase Storageバケット作成が「migrationファイルを書く→誰かが本番適用するまで機能しない」というギャップは、写真機能(`014_construction_photos_storage.sql`)の時から変わっていない構造的な制約。バケット作成をCLI/MCP経由で安全に自動適用できる仕組みがあれば、今回のような「実装は完了しているが本番では未稼働」の状態を減らせる。
