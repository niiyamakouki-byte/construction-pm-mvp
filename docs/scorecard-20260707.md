## 個別所見

### #3 PDFビューア（最重点検証・2026-07-07修正済み）
- **根本原因**: `DocumentsPageImpl.tsx`の`getPreviewConfig()`が直接.pdf URLを素の`<iframe src={url}>`でブラウザネイティブPDFビューアに丸投げしていた。自動化/一部環境で描画に完全失敗し黒一色になり、通常表示でも420px程度の小さい枠ではネイティブビューアのレターボックス背景が黒優勢になりやすい実装だった。
- **修正**: 新規`src/components/PdfCanvasPreview.tsx`で、既存依存の`pdfjs-dist`(4系、`pdf-to-estimate`機能で既に使用中でバンドル追加コストほぼ0)を使い、PDFページを自前`<canvas>`へ描画する方式に置き換え。ズーム(±ボタン+2本指ピンチ)・回転(90°刻み)・ページ送りを追加実装。読み込み失敗時は黒画面ではなくエラーメッセージにフォールバック。Google Drive/Docsの`/preview`埋め込み(黒画面の原因ではなかった経路)は無変更。
- **検証**: ローカルdevサーバー+`window.__E2E_BYPASS_AUTH__`(LocalStorageRepository経路、本番Supabase非接触)でテスト用PDF(`https://mozilla.github.io/pdf.js/web/compressed.tracemonkey-pldi-09.pdf`)を登録し、chrome-devtools MCPで実機確認。
  - **デスクトップ1440px**: 本文テキストが正常描画。スクショ: `~/reports/genbahub-pdfviewer-fix-20260707/01_documents_pdf_preview_desktop_1440.png`
  - ページ送り・拡大(125%)・回転操作も実機で確認: `~/reports/genbahub-pdfviewer-fix-20260707/02_documents_pdf_preview_desktop_page2_zoom_rotate.png`
  - **モバイル390px**: 同様にPDF本文が正常描画。スクショ: `~/reports/genbahub-pdfviewer-fix-20260707/03_documents_pdf_preview_mobile_390.png`
- **テスト**: `PdfCanvasPreview.test.tsx`(3件、黒画面バグの回帰テスト含む)・`DocumentsPageImpl.test.tsx`への回帰テスト追加(1件)。既存スイート含め全565ファイル/8089件が緑。
- bead `construction_pm_mvp-6jt` クローズ済み（黒画面バグ、`14949c6`）。

### #3 追撃修正: 競合/回転バグ (`28732a7`)
- fable1によるコミット`14949c6`のレビューで実バグ2件を発見:
  1. `getViewport({rotation})`が常にページ自身の`/Rotate`を上書きし、埋め込み回転情報を持つPDFが初回描画から横倒しになっていた。
  2. ピンチ操作による急速なscale変更で、前回`render()`タスクの`cancel()`が実際に解放される前に同一canvasへ次の`render()`が発火し、pdf.jsの「same canvas」例外でビューア全体がエラー画面に落ちていた。
- **修正**: viewport算出をpage自身の`rotate`とユーザー回転の合成に変更。`renderPage`を世代カウンタでシリアライズし、前タスクの完了を待ってから次を描画するように変更。
- **テスト**: 修正前に失敗・修正後に成功することを確認した回帰テストを`PdfCanvasPreview.test.tsx`に追加（pdf.jsの実際の非同期cancelタイミングとbusy-canvas例外を模したフェイクrenderタスクを使用）。
- 本注記は laporta-beads-uvfp（スコアカード#3行の修正済み反映）に対応。

### モバイル既知の粗2件 修正（2026-07-07 追加ループ、`aac1393`）
検証環境: ローカルdev(`vite --port 5183`)+`window.__E2E_BYPASS_AUTH__` 及び本番 https://construction-pm-mvp.vercel.app/ を chrome-devtools MCP でモバイル実機相当(390x844, emulate mobile+touch)で確認。

- **A. ドキュメント/PDFビューアへの導線が見つからない**
  - **根本原因**: `ProjectListPage.tsx`の案件カードは丸ごと1つの`<button>`で、タップすると通常モード案件は常に`/gantt/:id`へ直行（メモモード案件のみ`/project/:id`へ）。`ProjectDetailPage.tsx`のドキュメントタブ自体は既存実装済みで正しく動作するが、**通常案件では一覧画面からそこへ辿り着く経路が実質存在しなかった**（唯一の経路はTodayDashboardPageの「全案件俯瞰」＝エグゼクティブペルソナ限定セクション）。モバイルはデスクトップのような常設サイドバーが無い分、この欠落がそのまま「導線が見つからない」として顕在化していた。
  - **修正**: `ProjectListPage.tsx`の案件カードに「ドキュメント」ボタンを追加（`/project/:id/documents`へ直接遷移、`stopPropagation`でカード本体の工程表遷移とは独立）。カード外枠は`<button>`→`role="button"`の`<div>`に変更（ネスト不可な`<button>`内`<button>`を避けつつキーボード操作(Enter/Space)を維持）。
  - **検証**: ローカルdevでモバイル390px実機確認。ホーム画面カードから1タップで「案件ドキュメント」画面（ドキュメント検索・アップロード登録・既存タブ`概要/ドキュメント/チャット/収支`）に到達することを確認。回帰テスト`project-list.test.tsx`に追加（ドキュメントボタン押下で`/project/xxx/documents`へ遷移し`/gantt/`へは遷移しないことを検証）。

- **B. モバイルでダッシュボードのレスポンシブ崩れ**
  - **根本原因**: 幅390pxでの「今日のダッシュボード」自体のレイアウト崩れ（横スクロール・文字サイズ縮小等）は再現しなかった（`viewport`メタ・`transform`/`zoom`とも正常）。一方、モバイル限定(`viewportWidth < 768`)で表示される固定位置(`position` computed by`defaultPos()`)のAI秘書FAB(丸型フローティングボタン)が、スクロール中にカード右端の数値表示（例: 本日の入場者「0名」、天候バッジ「延期候補」等）に重なり隠す実害を本番で screenshot 確認（`~/reports/genbahub-mobile-fix-20260707/09_prod_cockpit_390.png`, `10_prod_fab_overlap_390.png`）。デスクトップではこのFAB自体を表示しないため再現しない、モバイル固有のレスポンシブ不具合。
  - **修正**: `AssistantChatPanel.tsx`にスクロール検知(`isScrolling`)を追加し、スクロール中はFABの不透明度を100%→40%にフェード、スクロール停止500ms後に復帰（`transition-opacity duration-300`）。位置・サイズ・ドラッグ機能は無変更。
  - **検証**: ローカルdevで`window.scrollBy`実行直後にFABの`className`が`opacity-40`へ切替、700ms後に`opacity-100`へ復帰することをスクリプトで確認。スクロール中のスクリーンショット(`15_fab_faded_during_scroll_390.png`)で背後のバッジ文字が透けて読めることを確認。
  - **正直な限定事項**: 「縮小表示」という表現が指す事象がFAB重なりと同一かは未確定（レイアウト自体の縮小は本ループでは再現できなかった）。今回はscreenshotで実証できた「モバイル限定でダッシュボードの数値が隠れる」不具合をB相当として修正。もし別の縮小表示事象が再発するようなら、報告時の実機スクショを添えて再起票してほしい。
