# オンボーディングフロー設計（1画面3ボタン原則）

作成日: 2026-07-05
対象: GenbaHub（仮称）新規登録 → 初案件作成 → 現場メンバー招待

## 原則: 1画面3ボタン

各画面のクリック可能な選択肢は最大3つまで。内訳は次の形に固定する。

- **主ボタン 1つ**（塗りつぶし・大きめ・画面右下 or 中央）: 迷ったらこれを押せば前に進む
- **副ボタン 最大2つ**（枠線 or テキストリンク・控えめ）: 「戻る」「あとで」など主ボタンより弱い操作

我妻専務のようなアナログ層が「何を押せば次に進めるか」を一瞬で判断できることを合格基準とする。
主ボタンは常に同じ位置（画面右下）・同じ見た目（`bg-brand-500` 系の塗りつぶし）に統一する。

## 現状の実装（既存・変更なし）

`src/components/OnboardingWizard.tsx`（4ステップ、実装済み）:

| Step | 画面 | 主ボタン | 副ボタン |
|---|---|---|---|
| 1 | ようこそ | サンプル工程表を開く（amber、任意操作） | はじめる→（実質の主導線） |
| 1 | ようこそ | はじめる→ | （スキップなし、副ボタンなし） |
| 2 | 会社情報+最初の案件 | 次へ→ | 戻る / スキップ（計2つ）|
| 3 | 工事種別テンプレ選択 | 作成する→ | 戻る / スキップ（計2つ）|
| 4 | 完了 | 工程表を開く→ | なし |

Step1は「サンプルを開く」と「はじめる」の2択のみでボタン数原則を満たす。Step2/3は戻る・スキップ・次へで
ちょうど3つ。既存実装は本原則を(意図せず)満たしており、変更不要。

`src/components/OnboardingChecklist.tsx`（案件作成後にダッシュボードへ常設表示、実装済み）:
3ステップのチェックリスト（案件作成 / PDF見積を試す / 現場写真アップロード）。1ステップにつき
CTAボタン1つのみで原則を満たす。

## ギャップ: 「現場メンバー招待」は未実装

現行のオンボーディングには**メンバー招待の導線が存在しない**（`organization_members` テーブルは
スキーマ上あるが、招待UI・招待APIは未実装。コードベース全体を検索しても "招待" / "invite" の実装なし）。

7日間スプリントの本タスクでは招待機能の実装はスコープ外（doc追加が主）。以下は実装フェーズ向けの設計。

### 提案: Step4（完了画面）を拡張する（新規ステップは追加しない）

Simplicity First: 新しいウィザードステップを増やすと4→5ステップになり離脱が増える。
既存のStep4「準備完了！」画面に、招待への導線を**副ボタンとして1つだけ追加**する形にとどめる。

Step4の変更後イメージ:

- 主ボタン: 工程表を開く→（変更なし、既存動線を維持）
- 副ボタン: メンバーを招待する（テキストリンク、`/account` の招待セクションへ遷移）

これでStep4のボタン数は 1（主）+ 1（副）= 2 で原則内に収まる。

### 提案: 招待セクション（新規画面、`/account` 内に追加）

MVP版は**メール送信インフラなし**で成立させる（Simplicity First。現状 resend/sendgrid等の
送信基盤は未導入 — `grep`確認済み）。招待はリンク共有方式にする。

画面構成（3ボタン以内）:

- 主ボタン: 招待リンクをコピー（`organization_members`に`pending`招待レコードを作り、
  ワンタイムトークン付きURLを生成 → クリップボードへコピー）
- 副ボタン: 役割を選ぶ（admin / member の2択セレクト。ボタンとしては数えず入力欄扱い）
- 副ボタン: 完了して戻る

招待されたユーザーはリンクを開く → 既存のサインアップ画面 → トークンを検証して
`organization_members` に自動追加、という流れにする（メール配信を経由しないため、LINE/Slackで
リンクをそのまま貼れる = アナログ層でも運用できる）。

### 実装が必要な要素（未着手・実装フェーズでの参照用）

1. `organization_invites` テーブル（トークン、招待org_id、招待メール任意、role、有効期限、使用済みフラグ）
2. 招待リンク生成API（Supabase RPC or Vercel Function）
3. `/account` に「チームメンバー」セクションを追加（既存ページへの追加であり、リファクタではない）
4. サインアップ画面で招待トークンを受け取った場合、`ensure_user_organization` を呼ばず
   招待先organizationへ直接参加させる分岐（`OrganizationContext.tsx`の`ensureOrganization`に影響するため
   実装時は既存の自動組織作成ロジックと衝突しないよう設計レビューが必要）

## 全体フロー図（テキスト）

```
[新規登録画面]
   │ 主: サインアップ
   ▼
[OnboardingWizard Step1: ようこそ]
   │ 主: はじめる→   副: サンプルを開く
   ▼
[Step2: 会社情報+最初の案件]
   │ 主: 次へ→   副: 戻る / スキップ
   ▼
[Step3: 工事種別テンプレ選択]
   │ 主: 作成する→   副: 戻る / スキップ
   ▼
[Step4: 完了]
   │ 主: 工程表を開く→   副: メンバーを招待する(提案・未実装)
   ├─────────────┐
   ▼             ▼
[工程表画面]   [/account 招待セクション(提案・未実装)]
                  │ 主: 招待リンクをコピー   副: 役割選択 / 完了して戻る
                  ▼
              [招待された相手がリンクを開く]
                  │ 主: サインアップして参加
                  ▼
              [同じorganizationに参加、工程表が見える]
```

## 合格基準（検証可能な形）

- 新規登録から初案件作成まで、各画面のクリック可能ボタンが3つを超えないこと（現状実装で満たす）。
- 招待機能を実装する際も、招待画面のボタンが3つを超えないこと（本docの設計を満たす）。
- 我妻専務レベルのアナログ層が「今何をすればいいか」をラベルだけで判断できること
  （抽象的な英語ラベル・アイコンのみのボタンを主ボタンにしない）。
